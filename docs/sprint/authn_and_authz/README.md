# Authentication and Authorization Sprint

## Sprint goal

Deliver an end-to-end authentication and authorization foundation for tflive using Keycloak for authentication and user administration, and OpenFGA for resource-level authorization.

The completed sprint must let a platform administrator sign in using a secret-provisioned bootstrap account, create and manage users in Keycloak, assign global roles, and let authorized tflive users assign fixed roles on individual stacks. The API remains the source of enforcement; frontend visibility is only a usability aid.

## Completion outcomes

- Every non-health API endpoint requires a valid Keycloak access token.
- The API derives the actor from the token's immutable `sub` claim and never trusts browser-supplied actor identifiers.
- A platform administrator can manage users and global roles through the Keycloak Admin Console.
- Stack owners and platform administrators can manage per-stack access through tflive.
- Users can see and operate only the stacks allowed by their global and per-stack permissions.
- Stack templates, runs, logs, and artifacts inherit authorization from their owning stack.
- A requester cannot approve their own run, including when they are a stack owner or platform administrator.
- Authorization changes and sensitive denials are auditable.
- Authentication or authorization dependency failures fail closed.
- Production deployments have no embedded or default credentials.

## Scope boundaries

### In scope

- Keycloak-hosted sign-in, logout, token refresh, and user administration.
- Secret-backed bootstrap administration, with local-only development defaults.
- Global `platform-admin` and `stack-creator` roles.
- Fixed per-stack `owner`, `operator`, `approver`, and `viewer` roles.
- OpenFGA relationship storage, permission checks, accessible-object listing, and grant management.
- Single-tenant enforcement using the configured tflive tenant identifier.
- Backend, frontend, integration, security, audit, and operational documentation work needed for the functional foundation.

### Out of scope

- High availability, cross-region operation, and disaster-recovery automation for Keycloak or OpenFGA.
- Multi-tenant SaaS organization and realm design.
- Custom user-defined roles or arbitrary permission builders.
- Enterprise identity-provider federation, SCIM, and mandatory MFA policy.
- OPA-based contextual policies such as maintenance windows or network restrictions.
- A custom Keycloak login theme.

## Role model

| Scope | Role | Permissions |
|---|---|---|
| Global | `platform-admin` | Administer tflive and bypass ordinary stack checks, except self-approval protection. |
| Global | `stack-creator` | Create a stack and automatically become its owner. |
| Stack | `owner` | View, operate, approve, and manage access for the stack. |
| Stack | `operator` | View and operate the stack. |
| Stack | `approver` | View and approve eligible runs for the stack. |
| Stack | `viewer` | View the stack and its inherited resources. |

## Workflow

Tickets are ordered by their recommended execution sequence. A ticket may start only when every dependency is `Done`, unless the dependency is explicitly removed from the table during sprint refinement.

Permitted statuses are `Not Started`, `In Progress`, `Blocked`, and `Done`.

## Definition of done for every ticket

A ticket can move to `Done` only when:

- Its acceptance criteria are demonstrably satisfied.
- Relevant automated tests are added or updated and passing.
- Security-sensitive failure paths are tested, not only success paths.
- Configuration and operational behavior are documented where they changed.
- No secrets, access tokens, passwords, or usable production defaults are committed or logged.
- The backlog status and any dependency changes are updated in this file.

## Deferred follow-up sprint

High availability and disaster recovery intentionally remain outside this sprint. The follow-up operations sprint should cover replicated production topology, database backup and restore, realm and authorization-model recovery, multi-zone failure tests, capacity planning, upgrade rehearsal, and recovery-time and recovery-point objectives.
